VALMORA ENGINE: ARCHITECTURAL MANIFESTO & PLANNER CONTEXT
Core Philosophy and Vision
Valmora is not designed as a traditional, hardcoded Minecraft plugin; it is engineered as a highly modular, YAML-driven MMO/RPG engine running on PaperMC. The overarching goal of the architecture is to completely separate configuration, data definition, and dynamic scripting from the underlying Java implementation. Developers and server administrators should be able to create custom items, mobs, abilities, and user interfaces entirely through YAML configuration without altering the core codebase. To achieve this, Valmora relies on a decoupled architecture defined by four primary pillars: the Execution Engine (driven by the Execution Context), the Module Lifecycle system, the Unified Registry pattern, and a strictly isolated API layer. Every code modification, refactor, or feature addition must strictly adhere to these pillars to ensure the engine remains hot-reloadable, highly performant, and immune to memory leaks.
The Execution Engine and The Execution Context
At the very heart of Valmora’s dynamic behavior is the Execution Engine, which processes all custom mechanics, abilities, and scripted events. The central nervous system of this engine is the ExecutionContext. The Execution Context is a transient, immutable state container created at the exact millisecond a dynamic event occurs—such as a player swinging a custom sword, an armor piece triggering a passive effect, or a custom mob casting a spell. Rather than passing a fragmented list of variables (like the player, the entity hit, the location, and the item used) into various methods, the engine packages all of this environmental data into a single ExecutionContext object.
This design achieves absolute decoupling between the "Trigger" and the "Mechanic". A mechanic class, such as DamageMechanic or ApplyEffectMechanic, implements a single contract: void execute(ExecutionContext context). The mechanic does not know, nor does it care, how it was triggered. It simply interrogates the Context to ask for the caster, the target, the location, and the specific YAML parameters governing its behavior. Because the Context interface is standard, a single mechanic can be reused universally across custom items, mob attack patterns, GUI interactions, and future quest systems.
Furthermore, the Execution Context acts as a safe, intelligent parsing layer between raw YAML configurations and Java logic. Instead of mechanics reading directly from a Bukkit ConfigurationSection and risking type errors or null pointers, they request data through the Context using methods like context.getDouble("damage", 1.0). In the future, this abstraction allows the Execution Context to seamlessly interpret complex math expressions, dynamic placeholders (e.g., %player_strength% \* 2), and cross-mechanic variables without ever requiring a rewrite of the mechanic classes themselves. It transforms hardcoded listeners into a flexible, scriptable pipeline.
The Modular Lifecycle System
To ensure Valmora can be updated on the fly without restarting the entire server, it completely abandons global, monolithic initialization. Instead, the engine is divided into isolated domains such as the ItemManager, MobManager, CombatModule, PlayerManager, and UIManager. Every single one of these domains must implement the ReloadableModule interface. This interface mandates a strict lifecycle contract via onEnable() and onDisable() methods.
The ModuleManager orchestrates these lifecycles. When the server starts, or when a /valmora reload command is issued, the ModuleManager sequentially enables each module. Crucially, during a reload, the ModuleManager first calls onDisable() on every active module before calling onEnable() again. This means every module is entirely responsible for cleaning up its own state. If a module registers a Bukkit Event Listener, starts a Bukkit Runnable (like a regeneration task), or caches data, it must explicitly unregister those listeners, cancel those tasks, and clear those caches within its onDisable() method. Failure to do so will result in duplicated tasks, ghost entities (like orphaned damage indicator holograms), and severe memory leaks during hot-reloads.
The Unified Registry Pattern
Valmora strictly forbids the use of traditional static singletons (e.g., getInstance()) for data management, as they create rigid dependencies and complicate reloading. Instead, all loaded data definitions—such as custom items, mob types, and ability mechanics—are stored in generic Registries. The Registry<T> interface establishes a standardized contract for registering, unregistering, fetching, and clearing objects based on a unique string ID.
Implementations like SimpleRegistry<T> handle the thread-safe storage of these objects. When the ItemManager (acting as a Reloadable Module) starts up, it reads the YAML files, parses them into ItemDefinition objects, and registers them into an instantiated ItemRegistry. During a module reload, the manager simply calls clear() on the registry and re-parses the files. No system should ever bypass a registry to forcefully inject or retrieve data, and registries themselves must be instantiated locally within their respective modules, not held globally in static variables.
The API Abstraction Layer
To prevent the engine from collapsing into tightly coupled "spaghetti code", inter-module communication is entirely restricted to the ValmoraAPI. The main plugin class implements this interface and exposes it via a static provider. If the Combat Module needs to look up the stats of a custom armor piece, it does not fetch the concrete Valmora plugin instance or directly access the database. Instead, it queries ValmoraAPI.getInstance().getItemManager().
This API layer acts as a strict boundary. It ensures that internal implementation details are hidden from other modules, and it provides a safe, documented contract for external third-party plugins that wish to hook into the Valmora engine. Any refactoring or feature expansion must respect this boundary; systems must cast and request services solely through the defined API methods, ensuring that individual modules can be swapped, refactored, or disabled entirely without causing a cascading failure across the rest of the application.
